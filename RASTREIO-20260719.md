## 1) Migração para Supabase Auth real
Pedido do usuário: login local não bastava mais, precisava de autenticação de
verdade com isolamento de dados por professor.
- `src/supabase.js`: client via `@supabase/supabase-js`, funções
  `signUpWithEmail/signInWithEmail/signOut/getSession/onAuthStateChange/
  sendPasswordResetEmail/updatePassword`, mais `fetchCurrentState/
  saveCurrentState/fetchLatestSnapshots/fetchSnapshotById/createSnapshot/
  pruneOldSnapshots`, todas escopadas por `userId`.
- `supabase/schema.sql`: tabelas `diario_current` (1 linha por professor,
  `id = auth.uid()`) e `diario_snapshots` (histórico), RLS por `auth.uid()`.
- Decisões de produto (via `AskUserQuestion`, escolhidas pelo usuário):
  confirmação de e-mail **desligada** (acesso instantâneo), 1 diário por
  professor (não compartilhado), reset de senha por e-mail real do Supabase,
  PIN local no Android é só destrave rápido do app já logado — não é
  autenticação de verdade.
- `main.jsx`: estado de auth inteiro trocado — `session`, `authLoading`,
  drafts de login/signup/recovery, `appLockPinHash` (destrave por PIN,
  `loadAppLockPinHash/saveAppLockPinHash`), `appUnlocked = !!session`.
  Tela de login nova (split-screen, abas Entrar/Criar conta).

## 4) Bug crítico de perda de dados ao fechar/reabrir a aba
Sintoma relatado: fechar a aba e reabrir não pedia login e voltava pra "tela
de revisar arquivo" sempre, e uma edição real se perdeu uma vez.

**Causa raiz**: `loadData()` não desembrulhava o payload. O que fica salvo em
`localStorage[STORAGE_KEY]` é o payload completo de sincronização (com
`.data` aninhado + metadados de integridade), não o objeto de dados puro. Sem
desembrulhar, `migrateData()` não achava `classes` e devolvia estado vazio —
o app achava que local e remoto sempre divergiam, disparando o "revisar
arquivo" pra sempre. Fix: `migrateData(parsed?.data ?? parsed)`.

**Bugs irmãos encontrados junto** (mesma classe de problema — funções que
aplicavam dado mas esqueciam de persistir `localStorage`/hash local depois):
`applySyncReview` (revisão manual), branch "seguro pra aplicar automático" de
`loadLatestFromSupabase`. Corrigidos um por um.

## 5) Reescrita do salvamento: de "diário inteiro" pra "log de ações"
**Motivo**: qualquer edição salvava o diário inteiro (500KB+ em uso real) no
Supabase. As únicas garantias do navegador contra fechar a aba no meio
(`sendBeacon`, `fetch keepalive`) só cobrem até **64KB** — um payload desse
tamanho nunca cabe nessa garantia, não importa o quanto se ajuste o tempo de
espera. Risco real de perda silenciosa de nota/falta. Pedido explícito do
usuário: "vamos reescrever pra save ser por ação".

**Arquitetura escolhida** (documentada em
`C:\Users\falec\.claude\plans\glowing-shimmying-lightning.md`): manter o
diário como um documento único (JSONB), mas cada edição salva só um **patch**
pequeno (diff), não o documento inteiro. Alternativa descartada: reestruturar
em tabelas relacionais (mais "correto" a longo prazo, mas exigiria reescrever
~30 funções de edição — risco alto demais pra um app em uso real).

- **Motor de diff** (`main.jsx`, logo após `mergeBackupData`): `diffData`,
  `applyPatch`, `applyPatches`, `isPatchEmpty`, chaveado por `id` (ou chave
  composta) por campo-array conhecido (`classes`, `events`, `lessons`,
  `assessments`, `recoveries`, `attendanceSummaries`). **Importante**: a
  reconstrução é feita via `Map` por `id`, então aplicar o mesmo patch duas
  vezes é idempotente (não duplica) — isso permite reenviar/repetir ações com
  segurança em caso de retry.
- **Tabela nova `diario_actions`** (`supabase/schema.sql`): `user_id`,
  `created_at`, `patch jsonb`. RLS igual às outras. `supabase.js`:
  `appendAction` (usa `fetch` cru com `keepalive:true`, não o SDK — só assim
  dá pra marcar keepalive explicitamente), `fetchPendingActions(userId,
  sinceIso?)`, `deleteActions`.
- **`appendActionToSupabase()`** (`main.jsx`): calcula diff contra
  `lastSavedDataRef.current`, insere na `diario_actions`. Dispara a cada
  edição, com debounce de `AUTO_SYNC_DELAY_MS = 2000`ms.
- **`compactToSnapshot()`**: junta ações pendentes + estado local, grava um
  snapshot completo em `diario_current` (reaproveita `saveRemoteState`),
  apaga as ações já compactadas. Disparada por: `ACTION_COMPACTION_THRESHOLD
  = 20` ações acumuladas, `COMPACTION_INTERVAL_MS = 5min`, ou clique manual em
  "Salvar". **Busca as ações pendentes sem filtro de data** (não usa
  `sinceIso`) de propósito — descoberto que ações órfãs de uma compactação
  anterior (por causa de HMR duplo mount durante teste) ficavam presas pra
  sempre se filtrasse por "desde a última atualização".
- **`loadRemoteState`**: busca snapshot + ações pendentes desde
  `updated_at` do snapshot, aplica via `applyPatches` antes de comparar/
  mesclar com o estado local.
- **`saveToSupabase`** (botão manual "Salvar"): sem pasta escolhida, agora só
  chama `compactToSnapshot()` em vez de sobrescrever direto (evita apagar
  ações pendentes de outro aparelho que este cliente ainda não viu).

### Bug: closure desatualizada gerando "remoção fantasma"
Depois de implementar o log de ações, editar uma turma às vezes gerava DUAS
ações — um "add" e um "remove" espúrio do mesmo item, sem edição real no
meio. Causa: os listeners de `visibilitychange`/`pagehide` (flush ao
esconder aba) e de `online` (retry ao voltar conexão) tinham dependências de
efeito estreitas (`[appUnlocked]` / `[appUnlocked, autoSaveFailed]`, sem
`data`), então chamavam uma versão antiga (fechada sobre um `data` velho) de
`appendActionToSupabase`. Um disparo tardio dessa versão velha comparava
`data` desatualizado contra `lastSavedDataRef.current` já atualizado,
gerando um patch de "isso não existe mais localmente" (remove) espúrio.
**Fix**: `appendActionRef = useRef(appendActionToSupabase)` atualizado a
cada render; os listeners chamam `appendActionRef.current()` em vez da
função capturada diretamente. Verificado: adicionar turma + simular
`visibilitychange→hidden` na hora gera só UMA ação (add), sem remove
fantasma.
**Armadilha de teste**: rodar `npm run build` enquanto um `npm run start`
separado também faz hot-reload cria DOIS componentes montados na mesma aba
(HMR duplica `ReactDOMClient.createRoot()`), o que por si só já causa ações
duplicadas nos logs — sempre fechar abas antigas e abrir uma nova depois de
rebuild antes de testar isso.

## 6) Indicador de status de salvamento + bug do "sair do site?"
Pedido do usuário: já que não dá pra bloquear literalmente o navegador de
minimizar/fechar (não existe API pra isso), a mitigação real pedida foi (a)
indicador sempre visível de "salvo/salvando/falhou" e (b) reenviar sozinho ao
voltar a ficar visível se ainda estava pendente.
- Novo estado `hasUnsavedChanges` (+ `hasUnsavedChangesRef` p/ uso em
  listeners), ligado nos mesmos pontos que já mexiam em
  `autoSaveMessage`/`autoSaveFailed`: `true` ao agendar uma edição, `false`
  ao confirmar salvo (ou quando patch vazio), continua `true` em erro.
- Pill visual `.sync-status-pill` no cabeçalho (ao lado do botão Salvar):
  "Tudo salvo" (verde) / "Salvando..." (âmbar) / "Falha ao salvar" (laranja).
- `visibilitychange` pra "visible": se `hasUnsavedChangesRef.current` ainda
  for true, tenta salvar de novo sozinho (seguro por causa da idempotência
  do diff engine — ver seção 5).

### Bug encontrado pelo usuário em produção: aviso "Sair do site?" aparecendo com tudo salvo
Usuário reportou (com print do Chrome real) que o aviso nativo "É possível
que as alterações feitas não sejam salvas" aparecia mesmo com o pill mostrando
"Tudo salvo". **Causa raiz**: `autoSaveTimerRef.current` guardava o ID do
`setTimeout`, mas só era zerado no caminho de "flush ao esconder aba" — quando
o timer disparava **naturalmente** (sem minimizar nada), nunca era zerado.
Resultado: depois da primeira edição da sessão inteira, essa ref ficava presa
com um número antigo não-zero pra sempre, e o `beforeunload` (que checava
`!!autoSaveTimerRef.current`) achava que sempre havia algo pendente. **Fix**:
zera a ref dentro do próprio callback do `setTimeout` quando ele dispara
natural, e trocado o check do `beforeunload` pra usar `hasUnsavedChanges`
(estado real) em vez da ref (proxy que podia ficar dessincronizado). Testado
disparando `beforeunload` sintético via JS: `defaultPrevented=true` durante o
salvamento, `false` assim que confirma salvo.

## Armadilhas técnicas para sessões futuras (LLM ou humano)
- **Nunca usar PowerShell (`Get-Content -replace -Set-Content`) em arquivos
  com acentuação** — corrompe UTF-8. Usar a ferramenta `Edit`.
- **HMR duplica montagem** se `npm run build` roda enquanto `npm run start`
  também está de pé — sempre fechar abas de teste antigas e abrir uma nova
  depois de build antes de testar comportamento de estado/efeitos.
- **`git`/`gh` não estão no PATH** deste ambiente — usar o binário do GitHub
  Desktop (caminho no topo deste arquivo).
- **O motor de diff é idempotente por `id`** — reenviar a mesma ação (retry)
  nunca duplica dado, só sobrescreve a mesma chave. Isso permite retry
  liberal em qualquer código de sincronização sem medo de corrupção.
- **`diario_actions` deve ser varrida sem filtro de data** na compactação,
  senão ações órfãs de uma falha anterior ficam presas pra sempre.
- Sem chave `service_role` do Supabase disponível pro Claude — todo SQL
  precisa ser rodado manualmente pelo usuário no SQL Editor do painel.
- Ferramenta de screenshot do navegador interno é flaky (timeout comum) —
  preferir `get_page_text`/`javascript_exec`/`read_console_messages` pra
  verificação.
